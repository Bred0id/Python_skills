## Task 1

В далеком будущем, в 2157 году, образование стало полностью виртуальным. Студенты учатся в цифровых академиях, где их успехи отслеживаются с помощью продвинутых алгоритмов. Вы — разработчик системы управления успеваемостью студентов в одной из таких академий. Ваша задача — создать класс Student, который будет хранить информацию о студентах и их оценках, а также предоставлять методы для добавления оценок и расчета среднего балла. За каждый предмет студент может получить оценку от 1 до 10. Если средний бал студента по всем предметам `>=8`, студент получает стипендию.

Создайте класс `Student` для представления студента. У студента должны быть аттрибуты:

* `name` - имя
* `surname` - фамилия
* `age` - возраст
* `grades_of_subject` - список оценок по каждому предмету
* `is_grants` - наличие стипендии 

У класса должны быть следующие методы:

* `add_grade(subject, grade)` - добавляет оценку для `subject` и возвращает `True` при успешном добавлении или `
* `has_grants()` - информация о наличии стипендии 
* `average_grade_for_subject(subject)` - средний бал по конкретному предмету
* `average_grade()` - общий средний бал по всем предметам


## Task 2

Вы работаете в компании ***Lumon Industries***, где сотрудники проходят процедуру "разделения". У каждого сотрудника есть два состояния:
* Рабочее "я" — знает только о работе и ничего о личной жизни.
* Личное "я" — знает только о личной жизни и ничего о работе.

Сотрудники работают на "разделенном этаже", где их рабочие "я" выполняют задачи, не зная, что происходит за пределами офиса. Ваша задача — создать систему для управления сотрудниками, их состояниями и задачами.

Создайте систему для управления сотрудниками и их "разделенными" состояниями. У вас должны быть следующие классы:

Класс `Employee`:

Атрибуты:

* `name` (имя сотрудника, например, "Марк",  "Хелли" и тд).

* `work_memory` (рабочие воспоминания, список задач).

* `personal_memory` (личные воспоминания, список событий).

* `is_at_work` (находится ли сотрудник на работе, по умолчанию False).

Методы:

*  конструктор принимающий `name`
* `enter_work()` — сотрудник "входит" в рабочее состояние. Теперь он знает только о работе.

* `leave_work()` — сотрудник "выходит" из рабочего состояния. Теперь он знает только о личной жизни.

* `add_work_task(task)` — добавляет задачу в рабочие воспоминания, если сотрудник сейчас на работе. Возвращает `True` если задача добавлена, иначе `False`.

* `add_personal_event(event)` — добавляет событие в личные воспоминания, если сотрудник сейчас **не** на работе. Возвращает `True` если задача добавлена, иначе `False`.

* `get_current_memory()` — возвращает текущие воспоминания (рабочие или личные, в зависимости от состояния).

Класс `LumonOffice`:

Атрибуты:

* `employees` (список сотрудников на разделенном этаже).

Методы:

* `add_employee(employee)` — добавляет сотрудника в офис.

* `start_workday()` — все сотрудники "входят" в рабочее состояние.

* `end_workday()` — все сотрудники "выходят" из рабочего состояния.

* `assign_task(employee, task)` — назначает задачу сотруднику (только если он на работе).


## Task 3

Вы разрабатываете упрощенную версию системы управления версиями, похожую на **GitHub**. Каждый репозиторий хранит историю коммитов, где каждый коммит связан с предыдущим и следующим коммитом. Это позволяет легко перемещаться по истории коммитов вперед и назад. Для реализации этой функциональности вы решили использовать **двусвязный список**.

У вас должны быть следующие классы:

Класс `Commit`:

Атрибуты:

* `hash` (уникальный идентификатор коммита, например, "a1b2c3").

* `message` (сообщение коммита, например, "Исправлен баг").

* `prev` (ссылка на предыдущий коммит).

* `next` (ссылка на следующий коммит).

Методы:

__str__() — возвращает строковое представление коммита в формате: "[hash]: [message]".

Класс `Repository`:

Атрибуты:

* `head` (последний коммит в истории).

* `tail` (первый коммит в истории).

Методы:

* `add_commit(hash, message)` — добавляет коммит в конец истории.

* `remove_commit(hash)` — удаляет коммит по его хэшу.

* `get_commit(hash)` — возвращает коммит по его хэшу.

* `print_history()` — выводит всю историю коммитов.

* `revert_to_commit(hash)` — откатывает историю до указанного коммита (удаляет все коммиты после него).

* статический метод `generate_hash()` - генерирует случайный хэш для нового коммита. Длина хэша 6 символов, он состоит только из букв и цифр.

* метод класса `from_list(cls, commits)` - создает репозиторий из списка коммитов.
